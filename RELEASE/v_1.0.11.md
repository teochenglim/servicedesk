# v1.0.11 — Persona-Driven UX Redesign: Manager role, ServiceDeskAdmin + Sudo-as

Theme: the Manager and ServiceDeskAdmin persona layers of the redesign in [DESIGN/08_design_ux.md](../DESIGN/08_design_ux.md), built on top of the RBAC/stage-tracking/attachments/rich-text/Customer-view foundation shipped in [v_1.0.10.md](v_1.0.10.md). AI-assisted drafting and the Knowledge Base feedback loop are split out to [v_1.0.12.md](v_1.0.12.md) since they're a distinct enough subsystem (needs an LLM/framework decision) to track separately from these two role-focused phases.

## Already shipped (don't re-do — see v_1.0.10.md)

RBAC capability model, stage-tracking overlay + Progress Bar, `RoleAgent` + API-token auth, attachments (DB-blob storage, Customer-upload privacy), Toast UI Editor rich text.

**Customer view rebuild** ([DESIGN/08](../DESIGN/08_design_ux.md) §8.4), shipped in this cycle: full-width "+ Submit a ticket" button; plain-English progress bar variant (`stageProgress(t, now, plain bool)` in `internal/httpapi/stagebar.go` — separate stage names and label wording, no "Ack"/"Mitigate" surfaced) plus a quoted last external note under the bar; Close/Reopen always offered regardless of status (illegal transitions still legitimately rejected server-side by the existing state machine); Queue name, Labels, and Audit Trail hidden from the Customer-rendered ticket page (staff-only surfaces). Reused the existing shared `tickets_workspace.html` rather than forking a separate template — same decision as the Engineer screen below.

**Engineer-specific screen** ([DESIGN/08](../DESIGN/08_design_ux.md) §8.5) — decision, not built: two tabs ("My tickets" / "Available in my queues"), stage-age-first default sort, "Raise ticket for a user," lightweight Transfer picker were scoped but deliberately not built as a separate template. Every role still shares `tickets_workspace.html`, and existing `view=mine-assigned`/`view=my-queues` filters already cover the data-access need. Revisit only if a real per-persona layout need emerges.

**Service Desk Manager** ([DESIGN/08](../DESIGN/08_design_ux.md) §8.6), shipped in this cycle:
- **Configurable SLA targets**: `Queue.SLATargets` (JSON blob) evaluated by a new `internal/sla` package (`sla.Table`/`sla.Rule{Priority, Category, Minutes}`) rather than inline logic in `service` — rules are Priority-only (Category wildcard) for now, but the row shape already supports a per-category/incident-type matrix later with no schema change. `TicketService.Create` now computes `SLADueAt` from the ticket's queue+priority+category at creation time (previously unset in all non-demo data — this also satisfies `RELEASE/v_2.0.0.md`'s open "SLA due-date automation" item). Editable per queue on `/queues` (Manager-only), 4 number inputs (P1-P4 minutes).
- **`GET /manager` dashboard**: per-queue open/breaching counts, per-engineer active load (flags overloaded at team-average + 1, the simpler of the two thresholds the design doc suggested), aggregate MTTD/MTTA/MTTM/MTTR computed directly from ticket stage timestamps.
- **`GET /manager/activity`**: every open ticket, most-recently-updated first, each row showing only the latest note as a 3-line-clamped preview plus a Transfer control. `Ticket.UpdatedAt` (already GORM-managed on every mutation) doubles as "last activity" rather than a new column — the one gap was note-only activity, closed via a new `TicketRepo.TouchUpdatedAt` called from `NoteService.Add`.
- Nav entry + `GET /` role-based landing dispatch (Manager → `/manager`, everyone else → `/tickets`) — extended by ServiceDeskAdmin's own landing case below.

**ServiceDeskAdmin + Sudo-as** ([DESIGN/02](../DESIGN/02_design_roles_and_tenancy.md) §2.5, [DESIGN/08](../DESIGN/08_design_ux.md) §8.3), shipped in this cycle:
- **Sudo-as**: `auth.Claims` gained `SudoByID`/`SudoByUsername`; `Manager.IssueSudoToken` mints a JWT carrying the *target's* identity (Role/OrgID/UserID) with the real admin recorded as SudoBy — every existing RBAC/queue-membership/visibility check downstream works completely unmodified, exactly per the design's minimal-diff intent. New `internal/service/sudo.go` (`SudoService.Start`/`Stop`, `CapSudo`-gated, rejects sudo'ing into your own account). `POST /admin/users/{id}/sudo/start` stashes the admin's own token in a second cookie (`sd_return_token`) so `POST /admin/sudo/stop` restores it without a re-login — never a silent timeout, only explicit return.
- **Audit trail**: `EventLog.SudoByID` added; `TicketService.logEvent` (and the new `role_changed` event) now attribute every action to the sudo target's identity while separately stamping who's really sudo'd in, so a single row is self-describing. `GET /admin/audit`: filterable (actor ID, event substring, sudo-only) with a CSV export (`?format=csv`).
- **Persistent banner**: `baseData.SudoBy` (empty outside a sudo session) drives an always-visible "Acting as {target} ({role}) — sudo'd in by {admin}" bar in `layout.html`, with the return action inline.
- **ServiceDeskAdmin home reshape**: `GET /admin` is now the short DESIGN/08 §8.3 dashboard (user list with per-row Sudo-as/role-change, recent audit events, Org settings/Integrations/Audit buttons) — **and is now SystemAdmin-only** (previously any staff could view it as a flat nav-link list; the reshape exposes every user's role and one-tap sudo, so the gate was tightened from `agentOnly` to `adminOnly` as part of this change, caught via manual smoke-testing before it shipped). Manager's only lost path from that page (Custom Fields, gated on `CapQueueOps` not SystemAdmin) was added to the Manager dashboard instead.
- **Role changes**: `POST /admin/users/{id}/role` (previously-dead `UserRepo.UpdateRole` now wired up), audit-logged as `role_changed`.
- Landing dispatch: a bare SystemAdmin session now lands on `/admin`; a Manager (real or sudo'd-into) still lands on `/manager`.

**Note for whoever merges this**: this phase touches auth/RBAC/audit attribution — run `/security-review` before merging, per the original risk flag.
