## Shipped in this cycle

**AI-assisted drafting + AI Ticket Intelligence Panel core** ([DESIGN/08](../DESIGN/08_design_ux.md) §8.8–8.9):

- **LLM vendor seam**: new `internal/llm` package (`Client` interface + `HTTPClient`, speaking the OpenAI-compatible `/chat/completions` wire format, + `FakeClient` for tests) — one implementation covers a local Ollama (the default), OpenAI, Azure OpenAI, or any other self-hosted engine, just by changing `SERVICEDESK_LLM_BASE_URL`/`_MODEL`/`_API_KEY`. Mirrors the existing `EventPublisher`/`WebhookDispatcher`/`WorkflowTrigger` seam pattern.
- **Off by default**: `SERVICEDESK_AI_ENABLED` (default `false`) gates both the routes (unregistered entirely when off, matching the demo-reset route's pattern) and the UI (buttons/panel only render when `.AIEnabled`). Defaults to a local Ollama at `http://localhost:11434/v1`, model `qwen3:8b`. System prompts for both features are overridable via config (`ai.prompts.*` / `SERVICEDESK_AI_*_PROMPT` env vars) without a rebuild.
- **AI Ticket Intelligence Panel** (`internal/service/aisummary.go`, `TicketAISnapshot` model): regenerated wholesale on every new note (wired into `NoteService.Add`, run in a background goroutine so posting a note doesn't block on an LLM call), Engineer-facing only. Every generation is a versioned snapshot; a human edit (`POST /tickets/{id}/ai-summary/{field}/edit`) locks that field so the next auto-regeneration leaves it alone, and `POST .../regenerate` lets Engineer explicitly re-ask the model for a locked field, unlocking it. Verified end-to-end against a real local Ollama (`qwen3:8b`): a locked `diagnosis` field correctly survived a second note-triggered regeneration while other fields picked up new context.
- **AI-assisted drafting** (`internal/service/aidraft.go`): stateless, no persistence — `DraftDescription` (ticket submission, any authenticated user including Customer), `DraftResolutionNote`/`DraftTransferReason` (existing ticket, Engineer-only), all landing in the existing Toast UI Editor via a small `data-ai-url`/`data-ai-target` button convention in `app.js` — never auto-submitted.
- Fields shipped: `symptom`, `what_tried`, `problem_statement`, `diagnosis`, `mitigation`, `resolution` — the core panel from §8.9. See "Explicitly deferred" below for what's not in this set yet.

## Explicitly deferred (scoped out, not silently dropped)

- **Detected agenda items checklist + Suggest Split** (§8.9's bundled-asks splitting, §8.5's split trigger): needs a `TicketLink` table and per-item fulfilled/open tracking beyond the six scalar fields shipped above. Revisit once the core panel has real usage.
- **Mitigation-note path**: `TicketService.MarkMitigated`'s optional note (posted directly via `repo.NoteRepo`, bypassing `NoteService.Add`) does not currently trigger a panel regeneration, unlike notes posted through the normal composer. Minor gap, not wired to keep this phase's scope bounded.
- **Knowledge Base feedback loop** (§8.10) — moved to [v_2.1.0.md](v_2.1.0.md); a distinct subsystem on top of the panel above (which already produces the versioned snapshots the KB loop would promote from).

Not in scope: the existing Problems/RCA feature (`/problems`) is untouched.
