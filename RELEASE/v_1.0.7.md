# v1.0.7 — Thoughtworks brand UI redesign

Theme: visual and structural refresh of the existing HTML/HTMX frontend to the Thoughtworks brand (see [DESIGN/07_design_ui.md](../DESIGN/07_design_ui.md)). No new backend functionality, no data model changes beyond what's needed to render the new layout.

## Shipped

Built as specced below: brand tokens, persistent-pane tickets workspace (htmx `hx-select` swap), three-way conversation thread styling, segmented composer, responsive icon-rail/bottom-tab-bar and properties/bottom-sheet swap, all other pages re-themed. See "Explicitly deferred" for what's intentionally not included.

Two adjustments made against live feedback during implementation, beyond what the spec below literally says:
- The icon rail widened from 48px to 120px and each item shows an icon **and** a text label (not icon-only), per direct feedback while reviewing the rendered shell.
- `.app-main` needed its own `1.5rem` padding (the shell restructuring had dropped it), with the ticket workspace canceling it via a `-1.5rem` negative margin so the 3-pane layout still spans edge-to-edge.

## Scope decisions (confirmed with user before implementation)

- **Tickets workspace goes fully persistent-pane**, not just restyled. The list pane stays visible and keeps its scroll/selection state while the detail pane swaps via htmx — not two independent full-page navigations like today.
- **All other pages** (Queues, Problems, Admin, Login) get the new tokens/top-bar/icon-rail shell, but keep their current single-column card/table layout — no 3-pane treatment, that's ticket-specific.
- **Mobile**: full responsive layout (bottom tab bar replacing the icon rail, bottom sheet replacing the properties side panel) is in scope. Swipe-to-assign/snooze/close gestures are explicitly **out of scope** for this pass (would need custom touch-gesture JS with no physical device to verify against).
- Attach-icon and canned-response-icon in the composer (mentioned in DESIGN/07 §Composer) are **not implemented** — this repo has no attach/canned-response feature to hang them off, and inventing one is out of scope for a visual/structural pass.

## Design

### 1. Tokens & shell
- New `web/static/css/tokens.css`: the CSS custom properties block from DESIGN/07 §Brand tokens, verbatim. Loaded first in every page `<head>`, including `login.html` (which doesn't go through the shared `header` template block).
- `app.css` rewritten to consume the tokens instead of the current dark-neutral palette (`--bg`/`--panel`/`--accent`/etc.). Priority/status/SLA colors are their own fixed variables per DESIGN/07's explicit rule: brand color (teal/coral) and status semantics must stay visually separate.
- `layout.html` header/footer blocks restructured into: top bar (56px, `--tw-teal-900`, wordmark, centered search, notification bell, coral "New ticket" button, avatar+status dot) + icon rail (48px, desktop, `--tw-teal-900`, active item gets a 2px coral left bar). Rail items: Tickets, Queues, Problems, Admin (agents only), Profile/logout. Icons are small hand-authored inline SVGs — no icon font, no CDN, keeps the existing no-build-step constraint.
- Responsive breakpoint (matching the existing `.grid-2` 800px breakpoint convention in `app.css`) swaps the icon rail for a 56px bottom tab bar (5 icons max, active in coral, inactive teal at 60% opacity).

### 2. Tickets workspace (structural change)
- `handleTicketsList`'s query-building logic is extracted into a shared helper so `handleTicketDetail` can reuse it to populate the list pane too.
- `tickets_list.html` + `ticket_detail.html` merge into one `tickets_workspace` template made of two composable sub-templates: `ticket_list_pane` (left, 280px, unchanged filter/view controls) and `ticket_detail_pane` (right, fills remaining width; renders an empty state "Select a ticket to view details" when nothing is selected).
- `GET /tickets` renders the workspace with the list pane populated and the detail pane empty. `GET /tickets/{id}` renders the same workspace with both panes populated (detail handler now also runs the list query, default/unfiltered).
- Ticket rows get `hx-get="/tickets/{id}"`, `hx-select="#ticket-detail-pane"`, `hx-target="#ticket-detail-pane"`, `hx-swap="outerHTML"`, `hx-push-url="true"` — this is the same "fetch full page, select a fragment client-side" idiom `app.js` already uses for SSE-driven refresh of `#ticket-panel`, so no new server-side partial-response logic is needed. Direct links to `/tickets/{id}` still work as ordinary full page loads.
- List rows: 4px left bar in the priority color (Urgent red `#D64545`, High `--tw-amber-500`, Normal `--tw-teal-600`, Low `--tw-sage-500`), unread rows get a coral dot + bold subject, SLA chip on the right.
- SLA chip state (`ok`/`warning`/`breach`) computed from the existing `Ticket.SLADueAt` field via two new template funcs in `render.go` (`slaState`, `slaLabel`) — no schema change, `SLADueAt` already exists and is currently unused for anything but storage (per v1.0.0's deferred list).
- Ticket detail header: status pill + priority badge from the semantic set, assignee avatar, mid-teal used only for hover/link states, never as a fill.

### 3. Conversation thread & internal notes
- Notes render as three visually distinct types instead of today's single `.note` / `.note.internal` pair:
  - Customer author → left-aligned bubble, `--tw-surface-muted` background, rounded except bottom-left corner.
  - Agent/staff author → right-aligned bubble, `--tw-teal-600` at 10% tint, `--tw-teal-900` text, rounded except bottom-right corner.
  - `Internal` flag → full-width dashed-`--tw-amber-500` card (not a bubble), "Internal note" label + eye-off icon, pale amber background. This overrides bubble side/authorship styling entirely, so an internal note can never be visually confused with a customer-visible message.
- To tell customer vs. agent authorship: `ticketDetailData` gets a new `UserRoles map[int64]models.Role` field, built from the `allUsers` slice `handleTicketDetail` already fetches for the assignee dropdown — no new query.
- Timestamp + channel icon (email/chat/phone) in small caption text under each bubble. (Channel is not currently tracked per-note in the data model — this renders a generic icon rather than inventing a new field.)

### 4. Composer
- Segmented control ("Reply" / "Internal note") using Alpine.js (already loaded, `x-data`) for local toggle state.
- Reply active: `--tw-teal-600` underline, coral send button. Internal note active: `--tw-amber-500` underline, and the whole composer background shifts to the pale amber tint immediately on selection — the color warns the agent before they type, per DESIGN/07.
- Existing `internal` checkbox in `ticket_detail.html`'s note form is replaced by the segmented control driving a hidden `internal` form field; POST target/behavior (`/tickets/{id}/notes`) unchanged.

### 5. Other pages & copy
- Queues, Problems, Admin (index + all sub-pages), Login: apply `tokens.css` + the new top bar/icon rail shell (Login keeps its own centered-card layout, just re-themed, no rail since there's no authenticated nav).
- Fix the two em dashes found in current UI copy: `queues.html:24` ("No engineers assigned yet — nobody...") and `admin_orgs.html:32` ("Parent org ID (optional — lets a Department...")) — replaced with commas, per DESIGN/07's "no em dashes anywhere in UI copy" deliverable.

### 6. Docs
- `DESIGN.md`: add entry 7 linking `DESIGN/07_design_ui.md` (currently missing from the index).
- This file (`RELEASE/v_1.0.7.md`) gets flipped from spec to "Shipped" once implementation + `make vet test` + manual smoke test pass.
- No `VERSION` bump, no tag, no release — that stays the user's call per `CLAUDE.md`.

# UI Redesign Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reskin and restructure the ServiceDesk frontend to the Thoughtworks brand per DESIGN/07_design_ui.md, turning the ticket list/detail pages into one persistent-pane workspace, without changing any backend business logic.

**Architecture:** New `tokens.css` feeds brand CSS variables into a rewritten `app.css`. `layout.html` gains a top bar + icon rail (desktop) / bottom tab bar (mobile) shell. `tickets_list.html` + `ticket_detail.html` merge into one `tickets_workspace.html` template backed by one `ticketsWorkspaceData` struct; `GET /tickets` and `GET /tickets/{id}` both render it, and ticket rows use htmx (`hx-select` + `hx-swap=outerHTML`) to update only the detail pane in place — the same "fetch full page, select a fragment" idiom `app.js` already uses for SSE refresh. All other pages get the new shell/tokens but keep their existing single-column layout.

**Tech Stack:** Go 1.x (`html/template`, `net/http`), htmx (vendored, already in repo), Alpine.js (vendored, already in repo), plain CSS custom properties — no new dependencies, no build step.

## Global Constraints

- Never hardcode brand hex values in templates/CSS — always reference the `--tw-*` custom properties from `tokens.css`.
- Brand colors (teal/coral/sage/amber/purple) must never double as status/priority/SLA semantics — those use the separate fixed palette in DESIGN/07 §"Important: keep brand color separate from status semantics".
- No em dashes in any UI-facing copy (templates, error strings) — use a comma or full stop.
- No new JS dependencies, no build step — vanilla JS / Alpine directives already loaded / hand-authored inline SVG icons only.
- Every existing route, handler signature (except the two explicitly refactored in Tasks 6–7), and POST action must keep working exactly as before — this is a visual/structural pass, not a behavior change.
- `models.TicketStatus` values are `"New"`, `"In Progress"`, `"Resolved"`, `"Closed"`, `"Rejected"` (note the literal space in `"In Progress"` — the existing `status-{{.Status}}` class produces two space-separated class tokens for it, `status-In` and `Progress`, which is why the current `.badge.status-In-Progress` CSS rule silently never matches; Task 3 fixes this by keying off `data-status` attributes instead of relying on the raw class string). `models.Priority` values are `"P1"`–`"P4"` (not the spec's literal "Urgent/High/Normal/Low" — map P1→Urgent-red, P2→High-amber, P3→Normal-teal, P4→Low-sage, reusing the existing `prio-P1`..`prio-P4` class convention already in templates).
- Run `make vet test` after every task that touches `.go` files; run `make build` after every task that touches templates/CSS (confirms the embed + `html/template` parse still succeed).

```go
package httpapi

import (
	"fmt"
	"html/template"
	"net/http"
	"time"

	"servicedesk/internal/markdown"
	"servicedesk/web"
)

var funcMap = template.FuncMap{
	"markdown": markdown.Render,
	"add":      func(a, b int) int { return a + b },
	"assignedTo": func(assignee *int64, userID int64) bool {
		return assignee != nil && *assignee == userID
	},
	"slaState": func(due *time.Time) string { return slaStateAt(due, time.Now()) },
	"slaLabel": func(due *time.Time) string { return slaLabelAt(due, time.Now()) },
}

// slaStateAt buckets an SLA due time into "ok"/"warning"/"breach" for chip
// styling. Split out from the funcMap closure so it's testable without
// depending on wall-clock time.
func slaStateAt(due *time.Time, now time.Time) string {
	if due == nil {
		return ""
	}
	remaining := due.Sub(now)
	switch {
	case remaining < 0:
		return "breach"
	case remaining < 2*time.Hour:
		return "warning"
	default:
		return "ok"
	}
}

func slaLabelAt(due *time.Time, now time.Time) string {
	if due == nil {
		return ""
	}
	remaining := due.Sub(now)
	if remaining < 0 {
		return "Breached " + humanDuration(-remaining) + " ago"
	}
	return humanDuration(remaining) + " left"
}

func humanDuration(d time.Duration) string {
	d = d.Round(time.Minute)
	h := d / time.Hour
	d -= h * time.Hour
	m := d / time.Minute
	if h > 0 {
		return fmt.Sprintf("%dh %dm", h, m)
	}
	return fmt.Sprintf("%dm", m)
}
```

```go
func (s *Server) handleTicketDetail(w http.ResponseWriter, r *http.Request) {
	claims := middleware.ClaimsFrom(r.Context())
	id, err := strconv.ParseInt(r.PathValue("id"), 10, 64)
	if err != nil {
		http.NotFound(w, r)
		return
	}
	t, err := s.ticketSvc.Get(id)
	if err != nil {
		http.NotFound(w, r)
		return
	}
	if claims.Role == models.RoleCustomer && t.CreatorID != claims.UserID {
		// Same-org tickets they've been added to watch are visible too
		// (multi-tenant: "unless added by others within the same company").
		watching, wErr := s.watchers.IsWatching(id, claims.UserID)
		if wErr != nil || !watching || t.OrgID != claims.OrgID {
			http.Error(w, "forbidden", http.StatusForbidden)
			return
		}
	}

	tickets, queues, f, view, err := s.loadTicketsList(r)
	if err != nil {
		s.log.Warn("ticket detail: could not load ticket list pane", "err", err)
	}

	includeInternal := claims.Role != models.RoleCustomer
	notes, _ := s.noteSvc.ListForTicket(id, includeInternal)
	events, _ := s.events.ListForTicket(id)
	tags, _ := s.tags.ListForTicket(id)
	watching, _ := s.watchers.IsWatching(id, claims.UserID)
	waiting, _ := s.workflowTask.ListWaitingForTicket(id)
	approvals, _ := s.approvals.ListForTicket(id)

	waitingForms := map[int64]waitingForm{}
	for _, task := range waiting {
		wf, err := s.workflows.Get(task.WorkflowID)
		if err != nil {
			s.log.Warn("ticket detail: workflow lookup failed for waiting task", "task_id", task.ID, "workflow_id", task.WorkflowID, "err", err)
			continue
		}
		var cfg workflow.Config
		if err := json.Unmarshal([]byte(wf.Config), &cfg); err != nil {
			s.log.Warn("ticket detail: invalid workflow config", "workflow_id", wf.ID, "err", err)
			continue
		}
		if task.StepIndex < len(cfg.Steps) && cfg.Steps[task.StepIndex].Type == "user_input" {
			waitingForms[task.ID] = waitingForm{WorkflowName: wf.Name, Fields: cfg.Steps[task.StepIndex].Fields}
		}
	}

	allUsers, err := s.users.List()
	if err != nil {
		s.log.Warn("ticket detail: could not load users", "ticket_id", id, "err", err)
	}
	userNames := make(map[int64]string, len(allUsers))
	userRoles := make(map[int64]models.Role, len(allUsers))
	for _, u := range allUsers {
		userNames[u.ID] = u.Username
		userRoles[u.ID] = u.Role
	}

	var agents []models.User
	var workflows []models.Workflow
	if claims.Role.IsAgent() {
		for _, u := range allUsers {
			if u.Role.IsAgent() {
				agents = append(agents, u)
			}
		}
		wfs, _ := s.workflows.List()
		for _, wf := range wfs {
			if wf.IsRunbook && wf.Active {
				workflows = append(workflows, wf)
			}
		}
	}

	s.render.Render(w, "tickets_workspace", ticketsWorkspaceData{
		baseData: s.base(r, "Ticket #"+strconv.FormatInt(t.ID, 10)),
		Tickets:  tickets, Queues: queues, Filter: f, View: view,
		Ticket: t, Notes: notes, Events: events, Tags: tags, Agents: agents,
		Watching: watching, Workflows: workflows, WaitingTasks: waiting, WaitingForms: waitingForms,
		Approvals: approvals, UserNames: userNames, UserRoles: userRoles,
	})
}
```
