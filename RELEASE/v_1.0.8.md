# v1.0.8 — Demo mode

Theme: make a fresh ServiceDesk instance demo-able in under a minute — a config-gated mode that seeds realistic orgs/queues/engineers/tickets so a prospect or new hire sees a lived-in helpdesk instead of empty tables, without touching how the app behaves for real deployments.

## Shipped

Built as specced below, with two corrections against the original design made during implementation (both noted inline where they apply): seeding must run *after* `auth.Bootstrap`, not before, or it steals the reserved system-actor's user ID 1; and demo-row identification for reset uses the seeder's own fixed names/prefixes rather than a separate tag. No config file is required for any of this — `-demo`/`DEMO_MODE`, `-demo-reset`/`DEMO_RESET`, and `-seed-demo`/`SEED_DEMO_ONLY` all work as plain env vars or CLI flags with no YAML needed, consistent with every other setting in this project (see README.md's Configuration section).

## Design

### 1. Enabling demo mode

New `Config` fields in `internal/config/config.go`, following the existing env-var + YAML + CLI-flag pattern (see `StaticUsers`/`LDAPEnabled` for precedent):

| Field | Env var | YAML | CLI flag | Default |
| :--- | :--- | :--- | :--- | :--- |
| `DemoMode` | `DEMO_MODE` | `demo_mode` | `-demo` | `false` |
| `DemoReset` | `DEMO_RESET` | `demo_reset` | `-demo-reset` | `false` |
| `SeedDemoOnly` | `SEED_DEMO_ONLY` | (CLI/env only, not a persistent deployment setting) | `-seed-demo` | `false` |

- `DemoMode=true`: on startup, after `auth.Bootstrap` (never before — `Bootstrap` relies on the reserved "system" actor landing on user ID 1 as the very first row in the table; seeding first would steal that ID), run the seeder if the `organizations` table is empty. Non-demo deployments (`DemoMode=false`, the default) take none of this code path — zero behavioral change.
- `DemoReset=true` (only meaningful alongside `DemoMode`): wipe all seed-owned tables and reseed on every boot, for automated/ephemeral demo environments (e.g. a nightly-reset k8s deployment). Without it, a demo instance seeds once and then behaves like a normal server across restarts, preserving whatever the presenter did in the last demo.
- `SeedDemoOnly=true`: `main.go` runs the seeder against the configured DB and exits before starting the HTTP server — for pre-seeding a shared demo DB out-of-band, independent of the runtime `DemoMode` toggle. Mutually exclusive with actually serving traffic in that process invocation.
- A `make demo` target: builds and runs the binary with `DEMO_MODE=true` against the existing sqlite dev flow, for a one-command local demo.

### 2. Seed data

New `internal/demo` package, `Seed(db *gorm.DB, log *slog.Logger) error`, called from `main.go`. Idempotency (empty-table check, wipe-on-reset) lives in the caller, not in `Seed` itself — `Seed` always inserts.

Content, sized to look lived-in without being unwieldy to eyeball in a demo:

- **3 organizations**: `Acme Corp`, `Globex Inc`, `Initech` — customer orgs, per existing multi-tenant model.
- **2 queues**: `Service Desk` (general/Tier-1-shaped intake), `Network Ops` (specialized) — exercises queue-membership-gated pickup vs. QueueAdmin+ transfer.
- **Users** (created the same way `auth.Bootstrap` creates static users — `UserRepo.Create`/`UpsertStatic` with a known password hash):
  - 1 `QueueAdmin` (`demo.admin` / `demo1234`)
  - 4 `Engineer`s, distributed across the 2 queues via `QueueMembership` (`demo.engineer1..4` / `demo1234`)
  - 6 `Customer`s across the 3 orgs via `OrgMembership` (`demo.customer1..6` / `demo1234`)
  - The existing `admin`/`admin123` SystemAdmin bootstrap is untouched — demo mode adds users, it doesn't replace the default admin path.
- **~15 tickets** spread across all four statuses (New/In Progress/Resolved/Closed) and all priorities, assigned across both queues/engineers:
  - `SLADueAt` set so all three SLA-chip states are represented: a few already breached (past), a few in the warning window (soon), the rest comfortably ok (future) — see `slaState` in `render.go`.
  - A handful with note threads mixing customer, agent, and internal-only notes, to populate the three-way conversation styling from v1.0.7.
  - At least one ticket per status/priority combination that's plausible (no "Closed + Urgent + just created" nonsense).
- **1 Problem record** linking the 3 network-outage tickets, to show Problem Management isn't empty either.
- **1 Runbook** (`auto_assign` + `add_note` + `notify` steps), startable from any ticket, so the automation surface isn't a dead feature during a demo.
- Seeded rows are identified for reset purposes by their fixed, hardcoded names/prefixes (the 3 org names, the 2 queue names, the `demo.` username prefix, the workflow name) rather than a separate marker column/tag — simpler than the tag-based approach originally sketched here, and just as deterministic since those identifiers are constants shared between the seeder and the reset code.

### 3. Manual reset

`POST /admin/demo/reset` — SystemAdmin-only, only registered on the router when `cfg.DemoMode` is true (absent entirely otherwise, so it can't be hit or discovered on a real deployment). Deletes every row matching the seeded org/queue/user/workflow identifiers above and everything that cascades from them (notes, watchers, ticket tags, approvals, workflow tasks, the Problem link), then re-runs `Seed`. Exposed as a "Reset demo data" button in the admin UI, gated the same way.

### 4. UI signal

A small "DEMO MODE" badge in the top bar (from the v1.0.7 shell), rendered only when `cfg.DemoMode` is true — passed into the template context the same way the current user/role already is. The "Reset demo data" button (§3) renders next to it for SystemAdmin users only.

## Scope decisions

- No new subcommand framework (cobra etc.) — `-seed-demo` is a flag on the existing `flag.NewFlagSet`-based `config.Load()`, consistent with how every other setting in this project is exposed.
- Demo mode never touches non-empty, non-demo data by default (`DemoReset=false`): safe to accidentally set `DEMO_MODE=true` against a real DB — it just no-ops the seed step.
- Seed data volume (3 orgs / 2 queues / ~15 tickets) is deliberately small: enough to demonstrate every feature surface (multi-tenancy, queues, SLA states, notes, Problems, workflow automation) without requiring a presenter to scroll through pages of filler.

## Explicitly deferred

- No demo-specific sample data for webhooks/integrations (would require a real external endpoint to deliver to; out of scope for a self-contained demo).
- No scripted "demo tour" / guided walkthrough UI — this ships the data and the mode toggle, not a product-tour overlay.
- No per-persona demo login picker on the login page (e.g. "Log in as Engineer") — presenters use the seeded credentials directly.
