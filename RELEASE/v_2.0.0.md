# v2.0.0 — Webhooks + MTTx metrics integration

Theme: close the loop on operational visibility — the KPIs named in [DESIGN/01_design_overview.md](../DESIGN/01_design_overview.md) §1.2 (MTTD/MTTA/MTTM/MTTR) get computed and surfaced, not just implied by the data model.

## Already shipped (don't re-do)

- Webhooks themselves — registration, signed delivery, retry/backoff — were part of the original scope (see [v_1.0.0.md](v_1.0.0.md)).
- **MTTD/MTTA/MTTM/MTTR computation** (v1.0.10): stamped directly at the stage-tracking overlay's write points (`internal/service/ticket.go`), exposed as `servicedesk_ticket_mtt{d,a,m,r}_seconds` Prometheus histograms, and surfaced as live current-value aggregates on the Manager dashboard (v1.0.11, `GET /manager`) — computed directly from ticket stage timestamps, not scraped from `/metrics`.
- **SLA due-date automation** (v1.0.11): `Ticket.SLADueAt` is now actually populated at ticket creation from queue+priority(+category) policy (`internal/sla` package, `Queue.SLATargets`), configurable per queue via `/queues`.

## Planned / not started

- **SLA-breach alerting**: nothing currently fires when a ticket breaches - the Manager dashboard's breach counts are pull-only (computed when the page loads), not push. Every other webhook event (`ticket.created`, `.status_changed`, `.assigned`, `.mitigated`, `.updated`, `.label_added`, `note.added.*`) fires on a state transition someone triggers; SLA breach is time-based, so this needs a small background poller (similar cadence to the existing webhook/workflow worker) scanning for tickets whose `SLADueAt` just passed, dispatching a new `ticket.sla_breached` webhook event exactly once per breach.
- **Webhook event coverage review**: re-check for any other missing trigger a monitoring integration would want, now that the SLA-breach gap above is identified.
- Possible: a small built-in dashboard page rendering MTTx *trends* over time (last N shifts), not just the current single-value aggregates the Manager dashboard already shows, for smaller self-hosted deployments that don't want to run an external Grafana/Prometheus stack.

The trend-chart item is explicitly optional ("Possible") — SLA-breach alerting is the one real gap left in this version's theme.
