# v2.0.0 — Webhooks + MTTx metrics integration

Theme: close the loop on operational visibility — the KPIs named in [DESIGN/01_design_overview.md](../DESIGN/01_design_overview.md) §1.2 (MTTD/MTTA/MTTM/MTTR) get computed and surfaced, not just implied by the data model.

## Already shipped (don't re-do)

- Webhooks themselves — registration, signed delivery, retry/backoff — were part of the original scope (see [v_1.0.0.md](v_1.0.0.md)).
- **MTTD/MTTA/MTTM/MTTR computation** (v1.0.10): stamped directly at the stage-tracking overlay's write points (`internal/service/ticket.go`), exposed as `servicedesk_ticket_mtt{d,a,m,r}_seconds` Prometheus histograms, and surfaced as live current-value aggregates on the Manager dashboard (v1.0.11, `GET /manager`) — computed directly from ticket stage timestamps, not scraped from `/metrics`.
- **SLA due-date automation** (v1.0.11): `Ticket.SLADueAt` is now actually populated at ticket creation from queue+priority(+category) policy (`internal/sla` package, `Queue.SLATargets`), configurable per queue via `/queues`.

## Shipped in this cycle

**SLA-breach alerting**: `internal/service/sla_breach.go`'s `SLABreachChecker` polls (same `Run`/`ProcessOne` shape as `webhook.Dispatcher`, wired into the same `WorkerPoolSize` worker pool in `cmd/servicedesk/main.go`) for non-Closed tickets whose `SLADueAt` has passed and dispatches a new `ticket.sla_breached` webhook event exactly once per breach. `Ticket.SLABreachNotifiedAt` marks the claim; `TicketService.Transition` clears it on reopen so a ticket that breaches again after being reopened alerts again instead of staying silently suppressed.

**Two pre-existing concurrency bugs found and fixed along the way** (not introduced by the above, but surfaced by live-testing it under the default `WorkerPoolSize=4`):

- `WebhookDeliveryRepo.ClaimNext` and `WorkflowTaskRepo.ClaimNext` both used a transaction-wrapped SELECT-then-blind-UPDATE-by-ID, which let every pool worker's SELECT observe the same still-eligible row before any UPDATE committed — confirmed live, the same webhook delivery went out 4 times under the default pool size. Fixed by re-checking the exact eligibility condition in a second UPDATE (by the candidate's known ID) and checking `RowsAffected`, so only the worker whose UPDATE actually executes first wins the claim.
- `WebhookDeliveryRepo.ClaimNext` had a second, more fundamental gap even after that fix: a claimed delivery stayed `status=pending` with its original (already-due) `NextRunAt` for the *entire* multi-second HTTP delivery attempt, so every other worker's poll tick re-claimed and redelivered it regardless of the guard above. Fixed by pushing `NextRunAt` forward by a claim lease (30s, comfortably longer than the dispatcher's 10s HTTP timeout) as part of the same guarded UPDATE — `MarkDelivered`/`MarkFailed` overwrite it appropriately afterward, and a crashed worker's lease just expires, letting another worker pick it up (the same at-least-once retry behavior this queue already provides on real failures).
- Regression tests added at `internal/repo/claim_test.go` (including true concurrent-goroutine tests, 8 callers racing one claimable row, asserting exactly one wins) plus `internal/service/sla_breach_test.go` and `internal/httpapi/integration_sla_breach_test.go`. Full suite re-verified with `go test -race`.
