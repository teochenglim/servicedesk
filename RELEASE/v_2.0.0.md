# v2.0.0 — Webhooks + MTTx metrics integration

Theme: close the loop on operational visibility — the KPIs named in [DESIGN/01_design_overview.md](../DESIGN/01_design_overview.md) §1.2 (MTTD/MTTA/MTTM/MTTR) get computed and surfaced, not just implied by the data model.

## Already shipped in v1.0.0 (don't re-do)

Webhooks themselves — registration, signed delivery, retry/backoff — were part of the original scope and are already built (see [v_1.0.0.md](v_1.0.0.md)). What's left for this version is the *metrics* half of the theme:

## Planned / not started

- **MTTD/MTTA/MTTM/MTTR computation**: derive these from `event_logs` timestamps (ticket created → first assignment = MTTA; created → first resolution = MTTR; etc.) and expose them as Prometheus metrics and/or a dashboard view.
- **Webhook event coverage review**: confirm every trigger a monitoring integration would want (e.g. SLA breach) has a corresponding event; add any missing ones.
- **SLA due-date automation**: set `Ticket.SLADueAt` from queue/priority policy at creation, and alert (webhook/notify) when it's breached — currently the field exists but nothing populates or enforces it.
- Possible: a small built-in dashboard page (`/admin/metrics` or similar) rendering MTTx trends without requiring an external Grafana/Prometheus stack, for smaller self-hosted deployments that don't want to run one.

Scope not yet fully decided — brainstorm before implementing, per this repo's usual process for new features.
