# v2.1.0 — Knowledge Base Feedback Loop

Theme: the layer that makes [v_1.0.12.md](v_1.0.12.md)'s AI Ticket Intelligence Panel compound over time — every resolved ticket becomes a candidate contribution to a living, external-facing Knowledge Base, and every new ticket can pull from anything already learned ([DESIGN/08](../DESIGN/08_design_ux.md) §8.10). Builds on the versioned `TicketAISnapshot` data already produced by v1.0.12 — a human curator can promote a good snapshot into a KB article manually from day one, before any smarter matching exists.

## Already shipped (don't re-do — see v_1.0.12.md)

LLM vendor seam (`internal/llm`), AI Ticket Intelligence Panel with versioned snapshots + human-edit-outranks-regeneration semantics, AI-assisted drafting. All off by default via `SERVICEDESK_AI_ENABLED`.

## Planned / not started

- **`KBArticle` model**: customer-facing fields (symptom, what the customer should observe, self-service steps, resolution) plus internal-only validation/diagnosis notes; `draft`/`published` status; linked to the originating `TicketAISnapshot`.
- **Article lifecycle** (`internal/service/kb.go`): ticket resolves → propose a KB draft (new article or diff against an existing similar one) from the panel's final extraction, fired via the existing `WorkflowTrigger`-style interface, not a direct service import (per `ARCHITECTURE.md`'s layering rule). Human curation gate — Engineer or ServiceDeskAdmin approves before anything publishes; nothing auto-publishes to a customer-facing surface.
- **Matching**: `MatchForSymptom` against the structured extraction (symptom + what-to-observe), not raw ticket text. Simplest first pass (keyword/embedding-lite) — smarter matching is an enhancement, not a prerequisite, per the design doc's own framing.
- **Surfacing**: at ticket submission ("This looks like it might be: {article}. Try these steps first?" with an easy "didn't help, file anyway" path) and at Engineer triage ("Similar past tickets: {article}" with validation/diagnosis notes surfaced immediately). Additive UI only — no structural changes to the Customer/Engineer screens themselves.
- **Curation UI**: approval-queue handler + `kb_review.html`.
- **Manager dashboard metric**: deflection-rate / KB-match-rate tile (deferred from `v_1.0.11.md`'s Manager dashboard), computable once a real backlog of curated articles exists.

## Verification checklist

- An unapproved draft must never reach a Customer-facing surface — the one true trust-boundary bug class here, test explicitly.
- Ticket resolve → propose fires exactly once per resolution.
- Live smoke test: resolve a ticket, approve the draft, confirm it surfaces at a subsequent similar ticket's submission and at Engineer triage.

Not in scope: the existing Problems/RCA feature (`/problems`) is untouched.
