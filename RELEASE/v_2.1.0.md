# v2.1.0 — Knowledge Base Feedback Loop + Service Catalog

Theme: the layer that makes [v_1.0.12.md](v_1.0.12.md)'s AI Ticket Intelligence Panel compound over time — every resolved ticket becomes a candidate contribution to a living Knowledge Base, and every ticket can be tied to the business service it impacts ([DESIGN/08](../DESIGN/08_design_ux.md) §8.10). Builds on the versioned `TicketAISnapshot` data already produced by v1.0.12 — a human curator can promote a good snapshot into a KB article manually from day one, before any smarter matching exists.

Field list and Service catalog design grounded in a research pass over KCS methodology (Issue/Environment/Cause/Resolution), ITIL known-error-database conventions (symptom, affected service, root cause, workaround, resolution), and CMDB/business-service-model conventions (criticality tiers, owner, support group) — see the field-by-field rationale in [DESIGN/08](../DESIGN/08_design_ux.md) §8.10.

## Already shipped (don't re-do — see v_1.0.12.md)

LLM vendor seam (`internal/llm`), AI Ticket Intelligence Panel with versioned snapshots + human-edit-outranks-regeneration semantics, AI-assisted drafting. All off by default via `SERVICEDESK_AI_ENABLED`.

## Shipped in this cycle

**Service catalog** (`models.Service`, `internal/repo/service.go`, `internal/service/service.go`): the business-service concept (e.g. "Mail Service"), distinct from `Queue` (which team routes a ticket) — `Name`, `Description`, `Criticality` (`Critical`/`High`/`Medium`/`Low`), `Status` (`active`/`deprecated`/`retired`), optional `OwnerID`, `SupportQueueID`, and a self-referencing `ParentID` (same pattern as `Organization`/`Queue`, for a future component-service hierarchy). CRUD is **SystemAdmin-only** (`/admin/services`, `admin_services.html`) — a system-configuration concern like Users/Webhooks/Workflows, not a day-to-day queue/SLA concern like Queue/CustomFieldDef. `Ticket.ServiceID` is an optional nullable FK, settable at submission (`ticket_new.html`) and correctable at triage (`POST /tickets/{id}/service`, Engineer+) — "unknown" is a normal, common state, never required.

**`KBArticle` model** (`internal/models/models.go`): split into customer-facing fields (`Symptom`, `WhatToObserve`, `SelfServiceSteps`, `Resolution`) and engineer/internal-only fields (`Environment`, `RootCause`, `ValidationSteps`, `ResolutionSteps`, `Workaround`, `BlastRadius`), plus lifecycle/provenance (`SourceTicketID`, `SourceSnapshotID`, `RelatedArticleID`, `CreatedByID`, `ApprovedByID`, `PublishedAt`) and an external-sync schema seam (`ExternalSource`/`ExternalID`/`ExternalURL`/`LastSyncedAt` — same extension-point pattern as `Attachment.StorageBackend`; no push client shipped this cycle, no target system named). `KBArticleService` is a many-to-many join to the Service catalog above — an article can name multiple impacted services. Criticality is deliberately *not* duplicated onto `KBArticle`; it's read live off the linked `Service` so it can't drift from the catalog.

**Article lifecycle** (`internal/service/kb.go`, `KBService`):
- `ProposeFromTicket` fires from `TicketService.Transition` (via the new `KBProposalTrigger` interface, same decoupling pattern as `AISummaryTrigger` — `internal/service/interfaces.go`) exactly once per resolution, including again after a reopen → re-resolve cycle. Seeds fields from the ticket's latest `TicketAISnapshot` when one exists; falls back to empty seed fields otherwise, so the loop works independent of `SERVICEDESK_AI_ENABLED`.
- `MatchForSymptom`: lowercased token-set Jaccard overlap across published articles' `Symptom`+`WhatToObserve` — simplest first pass per the design doc's own framing. Used at propose time to link a new draft to an existing similar article (`RelatedArticleID`) as a proposed update rather than a fresh one.
- Human curation gate (`internal/httpapi/kb_handlers.go`): `GET /kb/review` lists drafts (Engineer+), `POST /kb/{id}` edits fields, `POST /kb/{id}/approve` publishes. `KBService.CanView` is the one trust-boundary check — `Status == published || actor.Role.IsAgent()` — enforced at both `GET /kb` (list) and `GET /kb/{id}` (detail, 404 for a non-staff viewer of a draft).
- `GET /kb`: the published, Customer-safe browse/search surface.

**Demo data**: `internal/demo/seed.go` now seeds the 5-service catalog and two KB articles (one published, one draft) so `make demo`/`DEMO_MODE=true` isn't empty on either surface; `Reset`/`wipe` clean these up alongside the rest of the demo dataset.

## Explicitly deferred (scoped out, not silently dropped)

- **Submission-time and triage-time suggestion popups** (§8.10's "This might be: {article}" UX at ticket submission, and "Similar past tickets" at Engineer triage) — `MatchForSymptom` is built and unit-tested, but not wired into either screen yet. `GET /kb` (plain manual browse/search) is what ships in its place this cycle.
- **Real external KB sync** (Confluence/Zendesk/Guru/etc.) — schema seam only (`ExternalSource`/`ExternalID`/`ExternalURL`/`LastSyncedAt`); no target system was named, so no push client was built.
- **Manager dashboard deflection-rate / KB-match-rate tile** (deferred from `v_1.0.11.md`'s Manager dashboard) — meaningful once real usage of the loop above exists and/or the suggestion popups above are wired.

## Verification checklist

- An unapproved draft must never reach a Customer-facing surface — verified via `internal/httpapi/integration_kb_test.go`'s `TestKB_UnapprovedDraftNeverReachesCustomer` and live smoke-tested (draft 404s for a Customer login, absent from `/kb`'s list, visible immediately after approve).
- Ticket resolve → propose fires exactly once per resolution, including again after reopen → re-resolve — verified via `internal/service/kb_test.go`'s `TestKB_ProposeFromTicketTwiceCreatesTwoDrafts` and `internal/httpapi/integration_kb_test.go`'s `TestKB_ResolveProposesDraftExactlyOncePerResolution`.
- Service catalog CRUD is SystemAdmin-only — verified via `TestServiceCatalog_CRUDIsSystemAdminOnly` (Manager, despite holding `CapQueueOps`, gets 403).
- Live smoke test: resolve a ticket, confirm the draft appears at `/kb/review`, approve it, confirm it surfaces at `/kb` and `/kb/{id}` to a Customer login.

Not in scope: the existing Problems/RCA feature (`/problems`) is untouched.
