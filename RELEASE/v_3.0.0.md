# v3.0.0 — Quick-wins backlog

Theme: small, self-contained gaps left behind by earlier versions — each independently shippable, none requiring a design pass first. Consolidated after a sweep of every `v_1.0.*.md`/`v_2.*.md` "Explicitly deferred"/"Planned / not started" section to check what's actually still open (several were already resolved by later versions — e.g. v1.0.0's SLA-automation deferral was closed out by v1.0.11/v2.0.0 — and weren't repeated here). Larger initiatives needing real design work are tracked separately in [v_3.1.0.md](v_3.1.0.md).

Not carried forward (judged not worth a backlog slot): v1.0.8's demo-mode cosmetics (scripted tour UI, sample webhook data requiring a real external endpoint) and v1.0.9's README image polish — noted here so they're not silently dropped, but they're not real product gaps.

**Swipe gestures on mobile ticket list rows** (deferred since [v_1.0.7.md](v_1.0.7.md), originally "assign/snooze/close") was scoped out of this pass entirely and stays in the backlog untouched — its "snooze" action was never a real feature anywhere in the codebase (no field, no endpoint), so building the gesture would have meant building snooze first, a meaningfully bigger scope than the rest of this pass.

## Shipped in this cycle

- **Dynamic custom fields on the ticket-create form** (deferred since [v_1.0.0.md](v_1.0.0.md)): the category input triggers an htmx partial (`GET /custom-fields/for-category`) rendering whatever `CustomFieldDef` rows match, submitted as `cf_<name>` fields and stored in the existing `Ticket.CustomFields` JSON blob; values now also render read-only on the ticket detail page. Along the way, `CustomFieldDef` CRUD itself moved from Manager's `CapQueueOps` gate to SystemAdmin-only, matching the Service catalog precedent (`RELEASE/v_2.1.0.md`) — defining what data gets collected is a system-configuration concern, not day-to-day queue/SLA config.
- **Mitigation-note AI panel regeneration** (deferred since [v_1.0.12.md](v_1.0.12.md)): `TicketService.MarkMitigated`'s note bypassed `NoteService.Add` (the only place that fired `AISummaryTrigger.Regenerate`); `TicketService` now takes its own optional `AISummaryTrigger` and fires the same async, best-effort regenerate call.
- **Webhook event coverage audit** (deferred since [v_2.0.0.md](v_2.0.0.md)): 9 event names actually fire today — `ticket.created`, `ticket.status_changed`, `ticket.assigned`, `ticket.mitigated`, `ticket.updated`, `ticket.label_added`, `note.added.external`, `note.added.internal`, `ticket.sla_breached`. One asymmetry found and *deliberately left alone* this pass: `workflow.notify` is real-time (SSE) only — it never reaches a webhook, unlike everything else. `admin_webhooks.html`'s placeholder text now lists all 9 real events so admins aren't guessing from 3 examples.
- **Knowledge Base suggestion popups** (deferred since [v_2.1.0.md](v_2.1.0.md)): `KBService.MatchForSymptom` (already built, already unit-tested) is now live at both points DESIGN/08 §8.10 originally called for — an explicit "Check for existing solution" button at ticket submission (`POST /tickets/match-symptom`, matches against title+description since no AI panel exists yet at that point), and an automatic "Similar past tickets" block at Engineer/Manager/SystemAdmin triage (matched against the AI panel's `symptom` field when one exists). `KBMatchThreshold` (formerly an unexported constant) is now shared between `ProposeFromTicket`'s existing use and both new call sites.
- **MTTx trend chart** (deferred since [v_2.0.0.md](v_2.0.0.md), "Possible"): the Manager dashboard's MTTD/MTTA/MTTM/MTTR tiles now each carry a 14-day inline-SVG sparkline (`TicketRepo.ListResolvedBetween` + day-bucketed aggregation in Go, `internal/httpapi/manager_handlers.go`) — no chart library, no build step, consistent with this app's vendored-JS-only frontend; styled per the dataviz skill's stat-tile convention (muted trend line, accent-colored latest-value marker, drawn from this app's existing `tokens.css` brand palette).
- **Demo persona login picker** (deferred since [v_1.0.8.md](v_1.0.8.md)): the login page, gated on `DemoMode`, now offers one-click login as each of the 4 roles — Customer (`demo.customer1` @ Acme Corp), Engineer (`demo.engineer1`), Manager (`demo.admin` — labeled correctly, not "Admin"), and SystemAdmin (the separate README-documented `admin`/`admin123` bootstrap account, always present regardless of demo mode).
