# v3.0.1 — SystemAdmin is the entire servicedesk

Theme: two bugs found by live use of v3.0.0's admin surfaces — a navigation gap and a capability-model reversal — both about the same underlying problem: SystemAdmin couldn't actually reach or control everything a "SystemAdmin" should.

## Fixed in this cycle

**Admin navigation was too buried.** Every admin-owned resource (Orgs, Custom Fields, Service Catalog, Webhooks, Workflows, Audit) already had a route and a handler, but on `/admin` (the one hub page) they were listed as a single muted line of small text links below the fold, easy to miss entirely — functionally undiscoverable next to Users' full card treatment. `admin_index.html`'s "System configuration" card now surfaces all of them as equally-prominent buttons, plus a direct link to `/queues` and `/metrics` (Prometheus) — the two "hidden" surfaces that were reachable only by typing the URL.

**SystemAdmin didn't hold `CapQueueOps` (Queue CRUD, per-queue SLA targets, cross-queue assign/transfer, the Manager dashboard) directly** — reached only via a Sudo-as session, by original design (`DESIGN/02` §2.1.1, dating to `v_1.0.11.md`). Explicit product decision this cycle: **SystemAdmin is the entire servicedesk** — the unambiguous top of the role hierarchy, holding every capability unconditionally, no Sudo-as required. `Role.Can` (`internal/models/models.go`) now returns `true` for `RoleSystemAdmin` regardless of which capability is asked, before falling back to the existing per-role `capabilityRoles` map for everyone else. This required no route changes at all — every `queueAdminOnly`/`RequireCapability(CapQueueOps)` gate, the Manager-nav-link visibility check (`.User.CanQueueOps`), and the service-layer defense-in-depth checks (`TicketService.assign`, `QueueService`) all key off `Role.Can`, so they picked up the new behavior automatically.

**Sudo-as's purpose narrows, doesn't disappear.** It no longer exists to *grant* SystemAdmin a capability it otherwise lacks (there isn't one left). What it's still genuinely for: acting *as a specific other person's identity* for things that are per-user facts rather than capabilities — most concretely `QueueMembership` (self-pickup requires being an actual member of that queue; holding `CapQueueOps` lets SystemAdmin manage membership/SLA/transfer for everyone without personally being a member of every queue), and any case where an audit trail or scoped view should reflect a named person rather than "the admin, using admin powers." `DESIGN/02` §2.5 is rewritten accordingly.

## Also updated

- `DESIGN/02_design_roles_and_tenancy.md` §2.1.1/§2.2/§2.4/§2.5, `DESIGN/03_design_functional_requirements.md` §3.2, `ARCHITECTURE.md`'s request-flow example, and `CLAUDE.md`'s capability-model convention note — all previously documented the withheld-capability version of this rule as a locked decision; all now describe the reversal.
- Tests updated to match: `internal/models/models_test.go` (`TestRole_CanQueueOps`, `TestRole_CanAgentDetect`), `internal/httpapi/integration_manager_test.go` (`TestQueueSLA_ConfigurableAndAppliedAtTicketCreation`), `internal/httpapi/integration_tenancy_test.go` (`TestQueueMembership_GatesPickupAndAssignIsManagerOnly`), `internal/httpapi/integration_sudo_test.go` (`TestSudo_StartActAsAndStop`, docstring + assertions re-scoped to what Sudo-as still does).

Old release notes (`v_1.0.11.md` and others) describing the original withheld-capability design are left as-is — they're an accurate record of what shipped at the time; this file is the record of the reversal, not a rewrite of history.
