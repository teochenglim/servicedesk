# v3.0.2 — Admin sidebar submenu

Theme: one more discoverability follow-up to `v_3.0.1.md` — putting every admin resource one click away from *anywhere* in the app, not just as buttons on the `/admin` hub page a SystemAdmin has to navigate to first.

## Fixed in this cycle

**Admin links moved into the persistent sidebar.** The icon-rail's "Admin" entry (SystemAdmin-only) now expands into a nested submenu — Orgs, Users, Custom Fields, Services, Webhooks, Workflows, Audit, Metrics — rendered on every page, not just `/admin` itself. Each sub-link highlights active the same way top-level rail items already do. `web/templates/layout.html` (`icon-rail-sub` block) + a small CSS addition (`web/static/css/app.css`); no new routes, all existing handlers. Left off the mobile bottom-tabbar deliberately — the Admin tab there still opens `/admin`, whose "System configuration" card (from `v_3.0.1.md`) already lists everything as full-width buttons, a better fit for a narrow icon-only bar than a nested flyout.

**Stale copy fixed.** `/admin`'s "System configuration" card still told SystemAdmin to "use Sudo as ... to edit" Queue/SLA — true before `v_3.0.1.md`, false after it (SystemAdmin holds `CapQueueOps` directly now). Copy corrected in the same pass that added the sidebar links, so both fixes shipped together instead of leaving the wrong instruction live in between.

## Verification

- `go build ./... && go vet ./... && go test ./...` clean (120 tests, unchanged — this cycle is template/CSS only, no Go logic touched).
- Live smoke test: logged in as `admin`/`admin123`, confirmed the sidebar submenu renders on an unrelated page (`/tickets`, not just `/admin`), and that navigating to `/admin/users` correctly highlights "Users" as active within the submenu.
