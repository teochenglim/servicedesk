# v3.0.3 — Semgrep SAST CI fix

Theme: the "Security" GitHub Actions workflow ([run #28867497386](https://github.com/teochenglim/servicedesk/actions/runs/28867497386)) was failing on `main` at commit `3a1188c` (v3.0.2). One blocking Semgrep finding, no functional bug behind it — a false positive plus a documentation gap in how the CI failure surfaces.

## Fixed in this cycle

**False-positive Semgrep finding suppressed.** `sparklineSVG` (`internal/httpapi/manager_handlers.go:309`, added for the manager dashboard MTTx trend tiles) builds an inline SVG with `fmt.Sprintf` and returns it as `template.HTML`, which skips Go's auto-escaping. Semgrep's `go.lang.security.audit.net.formatted-template-string.formatted-template-string` rule blocks on any `Sprintf` result passed to `template.HTML()` on principle, since that's a real XSS pattern *if* user-controlled data reaches the template. Traced every argument at all 4 call sites (`manager_handlers.go:91-94`): `w`/`h` are package constants, `len(values)` and the computed `points`/`lastXY` coordinates are floats derived from ticket duration aggregates, and `accent` is always one of 4 hardcoded `"var(--tw-*)"` string literals — never user input. Suppressed with `// nosemgrep: go.lang.security.audit.net.formatted-template-string.formatted-template-string -- ...` directly above the `fmt.Sprintf` call, same pattern already used for the two open-redirect false positives in `problem_handlers.go`/`ticket_handlers.go` (both predate this release).

**Root-caused via local reproduction, not by reading CI logs.** The GitHub Actions log for the `Semgrep SAST` job only shows the SARIF-mode summary (`Findings: 1 (1 blocking)`) — no file/line/rule detail, because the workflow runs `--sarif --output semgrep.sarif` instead of the default text reporter. Reproduced locally with the exact same `--config` flags (`p/golang`, `p/sql-injection`, `p/secrets`, `p/owasp-top-ten`) against the same commit to get the human-readable finding. Also worth noting for next time: the generated `semgrep.sarif` lists all matches semgrep evaluates (3, in this case) regardless of whether they're suppressed — `jq '.runs[0].results[].suppressions'` is the fast way to tell a live blocking finding from an already-`nosemgrep`'d one without re-running the scan in text mode.
