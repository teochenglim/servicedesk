# v3.0.4 — Customer view fixes

Theme: a Customer walked through the ticket UI (screenshot below) and flagged four things that read as broken or half-finished. All four were real gaps, not misunderstandings — this cycle closes each one from the Customer's side of the product, not the internal one.

## Fixed in this cycle

**Queues and Problems removed from the Customer nav.** Both are internal routing/RCA concepts a Customer has no use for (DESIGN/08 §8.4 only ever specified Tickets/Knowledge Base for this persona) — the sidebar and mobile tab bar showed them anyway, since the nav templates gated everything else (Manager, Admin) by role but left these two ungated. `web/templates/layout.html` now hides both links for `Customer`, and `GET /queues`/`GET /problems`/`GET /problems/{id}` are now `agentOnly` server-side too (defense in depth — a Customer could previously reach either by typing the URL directly, getting a 403 now instead).

**Self-service Knowledge Base — confirmed already working, no change needed.** `GET /kb` has been Customer-accessible since `RELEASE/v_2.1.0.md`/`v_3.0.0.md` (the "customer-safe browse surface," per the comment at `router.go:179`); this cycle just verified it live rather than assuming.

**Close ticket / Reopen now only appear when they're a legal move.** DESIGN/08 §8.4 originally called for showing both buttons in every ticket state, on purpose — "never gated... never disabled-and-hidden as a trap." Real customer feedback overturned that: seeing a Reopen button on a ticket that was never closed reads as a bug, not as flexibility. `tickets_workspace.html` now shows Close ticket only when `Status == Resolved`, and Reopen only when `Status` is `Resolved` or `Closed`; Add note stays unconditional, since that one was never confusing. DESIGN/08 §8.4 is updated to record the reversal and why. The state machine (`internal/service/statemachine.go`) already rejected illegal transitions server-side either way — this only fixes what the UI *offers*.

**Watch now supports inviting a few other people, not just yourself.** The old Watch button only ever toggled the current viewer. `TicketService.WatchByEmails` (`internal/service/ticket.go`) resolves a comma/newline/space-separated list of emails against existing accounts (`UserRepo.FindByEmail`, new) and adds each as a watcher; a Customer actor can only invite people in the ticket's own organization (`OrgMembershipRepo.IsMember`), staff can invite anyone — matching the existing Customer-visibility rule (`DESIGN/02` §2.3) rather than inventing a new one. There's no "unregistered external email" watcher concept — that would need a schema change plus outbound email delivery wiring, scoped out as a bigger v3.1.0-style item; this only lets you add people who already have a ServiceDesk login. The ticket page's new "+ Notify other people about this ticket" field defaults to the viewer's own email pre-filled, and reports back which emails didn't resolve to an account instead of silently dropping them (`POST /tickets/{id}/watchers`, new route).

**Upload/note/watcher failures stay on the ticket page instead of navigating to a bare error page.** This was the concrete bug behind "upload button action failed" — a large `.docx` (or any file over the 10MB default cap) hit `r.ParseMultipartForm`'s size limit, which returned a raw `text/plain` 400 response the browser rendered as a whole new page, with no way back except the browser Back button. `redirectToTicketWithError` (`internal/httpapi/ticket_handlers.go`) now redirects back to `/tickets/{id}?error=...` instead, and `tickets_workspace.html` renders that as a dismissible-by-navigation banner at the top of the ticket. Wired into the attachment-upload path (too-large, no-file-chosen, disallowed-type, read-failure) and the new watcher-by-email path. Note posting failures got the same treatment; illegal-transition attempts (mostly a staff/API concern now that the Close/Reopen buttons are gated) intentionally kept the plain 400 response, since two existing tests depend on that status code being machine-checkable and no one reported it as a UX problem.

**Ticket Details panel is no longer empty for a Customer.** Every card in that panel (Labels, Assign/Transfer, Runbooks, Audit Trail) was gated to staff, so tapping "Details" on the Customer view opened a panel with nothing in it. It now shows a "Problem" card — title, category, priority, submission date, and the full description — restating what's already on the main pane in a structured, labeled form rather than a wall of muted text. The Conversation thread's existing engineer/customer bubble styling (right/left-aligned, distinct color) was already correct and needed no change; a composer for the Customer's own notes has been present since the `DESIGN/08` §8.4 redesign — the screenshot's confusion here was the empty Details panel making it feel like something was missing.

## Verification

- `go build ./... && go vet ./... && go test ./...` clean (120 tests, up from 117 — new: `TestAttachment_RejectsDisallowedType` rewritten for the redirect-with-banner behavior; `TestCustomerView_GatesCloseReopenByStatusAndHidesStaffJargon` replaces `TestCustomerView_AlwaysShowsCloseReopenAndHidesStaffJargon`, driving a ticket New → Resolved → Closed and checking button visibility at each stage).
- Live smoke test via `make demo`, logged in as `demo.customer1` (Acme Corp): confirmed Queues/Problems are gone from both nav surfaces and 403 on direct URL; confirmed ticket #1 (New) shows neither Close ticket nor Reopen and the Problem card renders in Details; added `demo.customer2@example.com` (same org) as a watcher via the new field and confirmed they could then see the ticket; attempted watching a nonexistent email and got the inline "No ServiceDesk account found for" banner; uploaded a `.exe` and a 15MB file and confirmed both landed back on the ticket page with a specific inline error instead of a blank page.

Not in scope: true external (unregistered) email watchers with outbound notification delivery — flagged above as a bigger, SMTP-dependent feature needing its own design pass, not a quick add.
