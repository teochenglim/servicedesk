# v3.0.5 — Ticket submission form (Customer-facing)

Theme: continuing the customer-feedback cycle from v3.0.4, the customer next flagged the ticket submission form (`GET/POST /tickets/new`) — Category was a bare free-text input with no structure or guidance, Queue was a required dropdown a Customer had no reason to understand, and there was no way to name other watchers until after the ticket already existed.

## Fixed in this cycle

**Category is now a real catalog, not free text.** New `Category` model (`internal/models/models.go`) — self-referencing `ParentID`, same pattern as `Organization`/`Queue`/`Service` — schema-ready for the requested 3-layer nesting (e.g. Hardware → Laptop → Screen), but only top-level (`ParentID == nil`) categories are shown/selectable at submission this cycle, per the "1 layer first" ask. `Ticket.Category` stays the plain string it always was — the catalog only backs the dropdown and template lookup, it's not a foreign key, so nothing about existing `CustomFieldDef.Category` string-matching changed. CRUD is SystemAdmin-only at `/admin/categories` (`internal/httpapi/category_handlers.go`, `web/templates/admin_categories.html`), same gate as the Service catalog and Custom Field Definitions. A guaranteed default row ("General", ID 1) is seeded on every fresh install (`seedDefaultCategory` in `internal/db/db.go`, mirroring `seedDefaultQueue`) so submission never has an empty dropdown before an admin configures anything.

**Category selection prefills Title/Description with an editable template.** Each `Category` optionally carries a `TitleTemplate`/`DescriptionTemplate`; picking it in the submission form's new dropdown writes that text straight into the Title input and the Description editor via `initCategoryTemplates` (`web/static/js/app.js`) — same mechanism the existing AI-draft buttons already use to write into the Toast UI editor (`mount.toastEditor.setMarkdown`). A category with no template (empty string) never touches the fields, so it can't clobber something the customer already typed. Still fully editable afterward — this is a starting point, never enforced.

**Queue is gone from the Customer's submission form.** A Customer never picks a Queue at all now — every ticket they submit lands in the guaranteed default Queue #1, and staff Transfer it afterward if it's routed wrong, the same mechanism already used for any other misrouted ticket. `handleTicketCreate` (`internal/httpapi/ticket_handlers.go`) falls back to Queue #1 whenever `queue_id` doesn't parse (i.e. wasn't submitted at all), without touching staff's path — Engineer/Manager/SystemAdmin still see and use the Queue dropdown on this same form when raising a ticket on behalf of a customer (DESIGN/08 §8.5).

**Impacted service — confirmed already optional and already last, no change needed.** Verified live rather than assumed.

**Optional watchers at submission.** The submission form now has the same "notify other people" email field v3.0.4 shipped on the ticket-detail page — reuses `TicketService.WatchByEmails` unchanged. Unlike the detail page's dedicated action (which shows an error banner for unresolved emails), a typo'd invite here is only logged, not fatal — it shouldn't block the ticket from being created.

**Confirmed already correct, no change:** ticket title/description display, the always-visible Customer note composer, and Engineer's Reply/Internal-note toggle for posting a customer-visible note — all verified live against the running app, all already matched what was asked.
