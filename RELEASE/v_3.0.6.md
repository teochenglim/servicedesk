# v3.0.6 — Markdown editor was never actually mounting

Theme: while verifying v3.0.5's category-template prefill, the customer reported the Description field on the ticket-submission form wasn't a real input at all — just a label with nothing under it. Root-caused to the vendored Toast UI Editor bundle itself, not the v3.0.5 feature: it was broken everywhere it's used (ticket submission, ticket-detail note composer), and had been since it was first vendored.

## Root cause

`web/static/js/toastui-editor.min.js` was fetched from `cdn.jsdelivr.net/npm/@toast-ui/editor@3.2.2/dist/toastui-editor.min.js` — the **npm package's bundler-target build**. That file's own UMD wrapper does `require("prosemirror-commands")`, `require("prosemirror-history")`, `require("prosemirror-state")`, etc. — it expects a real bundler (webpack/rollup) to resolve those imports at build time. Loaded as a bare `<script>` tag with nothing providing those globals, `new toastui.Editor(...)` throws `Cannot read properties of undefined (reading 'PluginKey')` / `toastui.Editor is not a constructor` immediately, and the mount `<div>` (`web/templates/markdown_editor.html`) just sits empty at 0 height — no error visible anywhere in the rendered page, only in the browser console.

Confirmed via headless-browser testing (Playwright) in complete isolation (a blank HTML file with only this one script tag, no htmx/Alpine/highlight.js involved) that this reproduces identically across Chromium, WebKit, and Firefox, and across every 3.x release back to 3.1.10 — ruling out a browser-specific regression or a bad version pin. It's the wrong artifact for script-tag-only usage, not a broken release.

The npm package's own `dist/` folder (checked via jsDelivr's file listing) only ships `toastui-editor.js` (the peer-dependency build above) and `toastui-editor-viewer.js` (view-only) — no self-contained bundle. The actually-correct artifact for drop-in `<script>` tag use with no build step is `toastui-editor-all.min.js`, published on Toast UI's own CDN (`uicdn.toast.com`), which bundles ProseMirror internally. Confirmed working with zero errors in the same isolation test.

## Fixed

- `web/static/js/toastui-editor.min.js` replaced with the correct self-contained `toastui-editor-all.min.js` (534KB vs. the old 350KB peer-dependency build — the size difference *is* the bundled ProseMirror). `web/static/css/toastui-editor.min.css` replaced with its matching CSS from the same source, so nothing is left mismatched between the two.
- Same vendoring approach as htmx/Alpine/highlight.js already used in this repo (download a static file, commit it, no build step) — just the correct artifact this time. No template, handler, or `app.js` changes needed; `initMarkdownEditors`'s `new toastui.Editor(...)` call was always correct, it just never had a working library underneath it.
- This fixes every markdown editor mount in the app at once: the ticket-submission Description field, the ticket-detail Conversation composer (Customer's own notes and Engineer's Reply/Internal-note toggle), and any other `markdown_editor` partial usage — not just the one the customer happened to notice.
- v3.0.5's category-template prefill (`initCategoryTemplates` writing into `mount.toastEditor.setMarkdown(...)`) now actually works end-to-end, since `mount.toastEditor` is finally a real editor instance instead of never being set at all.

## Verification

- `go build ./... && go vet ./... && go test ./...` clean (122 tests, no count change — this is a vendored static asset swap, no Go logic touched).
- Live verification via Playwright (headless Chromium) against `make demo`: the ticket-new page's Description editor now reports a real bounding box (was 0 height, now 220px) with a full formatting toolbar visible; selecting "IT Support" as the category prefills the editor with its description template, confirmed by reading `toastEditor.getMarkdown()` back; the ticket-detail note composer mounts correctly too (160px); zero `pageerror` events on either page, down from 3 (two `PluginKey` TypeErrors plus the resulting `toastui.Editor is not a constructor`).
- Visual screenshot taken of the fixed ticket-new form confirming the rendered toolbar, editor box, and prefilled description text.
