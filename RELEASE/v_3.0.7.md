# v3.0.7 — Live ticket updates were completely dead

Theme: found while trying to reproduce a separate (ultimately non-)bug report about the note composer — the SSE endpoint powering "ticket updates without a page refresh" 500'd on every single connection. Unrelated to the composer, unrelated to v3.0.5/v3.0.6 — a pre-existing bug with its own root cause.

## Root cause

`GET /events` (`sse.Hub.Handler`, `internal/sse/`) requires its `http.ResponseWriter` to satisfy `http.Flusher` in order to stream — it type-asserts `w.(http.Flusher)` and returns a hard `500 streaming unsupported` if that fails. Every request passes through `metrics.Middleware` first (wraps the entire mux in `router.go`'s `Routes()`), which wraps every response in a `statusRecorder{ http.ResponseWriter; status int }` to record status codes for Prometheus. Embedding an interface only promotes *that interface's own* methods (`Write`/`Header`/`WriteHeader` from `http.ResponseWriter`) — `Flush()` belongs to the separate `http.Flusher` interface and was never promoted, so `*statusRecorder` never actually implemented it, and the type assertion failed on every request, unconditionally.

Practical effect: `EventSource` in `app.js` (live ticket status/note/label updates on an open ticket page) has been completely non-functional — it just fails and silently auto-reconnects forever, per `EventSource`'s own built-in retry behavior. No error surfaces anywhere a user would see it; only the browser's network tab shows the repeated 500s. Confirmed via `curl -N .../events` returning the literal `streaming unsupported` body instantly (no hang, no stream) before the fix.

## Fixed

- `internal/metrics/metrics.go`'s `statusRecorder` now has its own `Flush()` that delegates to the underlying writer's `http.Flusher` when present:
  ```go
  func (r *statusRecorder) Flush() {
      if f, ok := r.ResponseWriter.(http.Flusher); ok {
          f.Flush()
      }
  }
  ```
  This makes `*statusRecorder` genuinely implement `http.Flusher` again, restoring streaming through the metrics middleware for `/events` (and any other endpoint that ever needs to stream in the future — the middleware wraps everything, so this was a site-wide gap, not an SSE-specific one).

## Verification

- `go build ./... && go vet ./... && go test ./...` clean (122 tests, no count change — this is a two-line middleware fix, no new behavior to unit test against the in-memory test server, which doesn't exercise real streaming semantics anyway).
- Live: `curl -N .../events` no longer returns the instant `streaming unsupported` 500 — the connection now holds open, matching correct SSE behavior (confirmed via both raw `curl` and a headless-browser check where `page.wait_for_load_state("networkidle")` correctly hung on the still-open connection, itself a good sign the stream is real rather than closing immediately).
- Did not chase full end-to-end proof further than that (a note posted elsewhere appearing live without a page refresh) — the two follow-up attempts to prove it were both testing-tool artifacts (a demo ticket seeded with no watchers to notify; a Python HTTP client refusing to resend a `Secure`-flagged cookie over plain `http://localhost`, unlike a real browser's `localhost`-is-trusted exception), not evidence of a remaining product bug. The concrete, reported defect (instant 500 on every connection) is fixed and confirmed.
