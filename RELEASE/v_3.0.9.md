# v3.0.9 — Ctrl-C took 15s to kill the server once SSE actually worked

Theme: side effect of the v3.0.7 fix. Once `GET /events` stopped 500ing on every connection, an open browser tab holding a live SSE stream turned a previously-instant `Ctrl-C` into a 15-second hang.

## Root cause

`http.Server.Shutdown` stops accepting new connections and waits for active handlers to return on their own — it never cancels an in-flight request's context. `sse.Hub.Handler` blocks in a `select` keyed on `r.Context().Done()`, which only fires when the client disconnects. Before v3.0.7, SSE 500'd instantly so no long-lived connection ever existed to hold the drain open; after that fix, any open tab keeps the handler alive, so `main.go`'s `httpSrv.Shutdown(shutdownCtx)` blocks for the full 15s timeout, logs `context deadline exceeded`, and only then exits. Repeated `^C` during the drain is a no-op — `signal.NotifyContext` funnels every SIGINT into the same context, there's no force-quit path.

## Fixed

- `internal/sse/hub.go`: `Hub` gained a `closing` channel and a `Close()` method that closes it. `Handler`'s `select` loop now also returns on `<-h.closing`.
- `cmd/servicedesk/main.go`: calls `hub.Close()` immediately after the shutdown signal is received and before `httpSrv.Shutdown`, so every live stream returns on its own instead of waiting out the timeout.

## Verification

- `make vet test` clean.
- Live: started the binary, logged in as `demo.admin`, held a `curl -N /events` connection open, sent `SIGTERM` to the server. Shutdown completed in ~5ms (down from the full 15s timeout + `context deadline exceeded` error beforehand).
