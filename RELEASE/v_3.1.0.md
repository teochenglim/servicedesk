# v3.1.0 — Larger initiatives (needs a design pass first)

Theme: the harder half of the [v_3.0.0.md](v_3.0.0.md) backlog sweep — deferred items from `v_1.0.*.md`/`v_2.*.md` that each need real design decisions before coding starts, not just implementation time. Do not start coding against any item below without first working through its open questions, same rule v3.0.0's predecessor placeholder always carried.

## AI/Agent collaboration (was v3.0.0's original scope, narrowed)

Originally "let AI agents work alongside human Engineers within the ticketing flow, rather than treating automation as fixed rule/runbook scripts." [v_1.0.11.md](v_1.0.11.md)/[v_1.0.12.md](v_1.0.12.md) answered part of this (triage suggestions via the AI Ticket Intelligence Panel, draft responses, `RoleAgent` audit attribution, the `internal/llm` vendor seam) — what's left:

- **Runbook Hook integration**: whether an agent acts through the existing workflow step interpreter (a new step type, e.g. `llm_call`) or needs a different integration point. No such step type exists in `internal/workflow/steps.go` today — every AI feature shipped so far (`aisummary.go`, `aidraft.go`, `kb.go`) is invoked directly from `TicketService`/`NoteService`, not from the Runbook engine.
- **Literal reuse of the `approval` workflow step type** for an agent's own actions — the propose-then-human-approves *shape* has shipped twice independently (AI panel's human-edit-outranks-regeneration rule, KB's curation gate), but neither goes through the `approval` step type; worth deciding whether to generalize now that two instances exist to generalize from.
- **Autonomous auto-resolve** of any ticket class — still just a stated direction, no design. Every AI/agent feature shipped to date is propose-only; this would be the first genuinely autonomous action in the system and needs its own guardrail design, not an extension of the existing patterns.

## Other larger deferred items

- **LDAP/OAuth2** (deferred since [v_1.0.0.md](v_1.0.0.md)) — env var parsed (`LDAP_ENABLED`), never wired to a real directory. Security-sensitive; needs a real design pass (which directory protocol(s) to support, credential storage, group-to-role mapping) before implementation, not a quick wire-up.
- **Organization hierarchy rollup** (deferred since [v_1.0.0.md](v_1.0.0.md)) — `Organization.ParentID` is schema-ready but nothing gives a Group-level user visibility into child-org tickets; touches the multi-tenant Customer-visibility rules (`DESIGN/02` §2.3) system-wide, not a localized change.
- **Per-note channel tracking** (email/chat/phone, deferred since [v_1.0.7.md](v_1.0.7.md)) — needs a real design decision: just a decorative column reflecting how the note *arrived*, or an actual multi-channel integration (inbound email-to-ticket, chat webhook, etc.)? The two have very different scope.
- **Canned responses** (deferred since [v_1.0.7.md](v_1.0.7.md), originally "attach/canned-response icons in the composer") — no backing feature exists; needs a template library data model plus an insert-into-composer UX, not just an icon.
- **Detected agenda items checklist + Suggest Split** (DESIGN/08 §8.9/§8.5, deferred since [v_1.0.12.md](v_1.0.12.md)) — needs a `TicketLink` table and per-item fulfilled/open tracking beyond the AI panel's six scalar fields, plus the AI extraction prompt/parsing logic to populate it.
- **Real external KB sync** (Confluence/Zendesk/Guru/etc., deferred since [v_2.1.0.md](v_2.1.0.md)) — the schema seam exists (`KBArticle.ExternalSource`/`ExternalID`/`ExternalURL`/`LastSyncedAt`); needs a target system named and its API/credential model designed before any client code.
- **Manager dashboard deflection-rate / KB-match-rate tile** (deferred since [v_1.0.11.md](v_1.0.11.md), restated in [v_2.1.0.md](v_2.1.0.md)) — genuinely blocked on this file's own KB suggestion-popup wiring landing first (v3.0.0): without a "suggestion shown" signal from those popups, there's no match-attempt event to compute a rate from.
